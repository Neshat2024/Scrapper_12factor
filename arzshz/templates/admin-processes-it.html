XII. Processi amministrativi

Eseguire attività di amministrazione/gestione come processi una tantum

LeFormazione del processoè la matrice di processi usati per eseguire le normali attività dell'app, ad esempio la gestione delle richieste Web, durante l'esecuzione. Separatamente, gli sviluppatori spesso desiderano eseguire attività amministrative o di manutenzione una tantum per l'app, ad esempio:

  • L'esecuzione di migrazioni di database (ad es.manage.py eseguire la migrazionein Django,Rake db:migratein Rotaie).
  • L'esecuzione di una console (nota anche comeREPLshell) per eseguire codice arbitrario o controllare i modelli dell'app rispetto al database attivo. La maggior parte delle lingue fornisce un REPL eseguendo l'interprete senza argomenti (ad es.pitoneoPerl) o in alcuni casi hanno un comando separato (ad es.IRB (Italiano)per Rubino,Consolle di binariper le rotaie).
  • L'esecuzione di script una tantum di cui è stato eseguito il commit nel repository dell'app (ad es.Script PHP/fix_bad_records.php).

I processi amministrativi una tantum devono essere eseguiti in un ambiente identico a quello dei normali processi di amministrazioneProcessi a esecuzione prolungatadell'app. Corrono contro unrilascio, utilizzando lo stessoBase di codiceeconfigurazionecome qualsiasi processo eseguito su quella versione. Il codice amministratore deve essere fornito con il codice dell'applicazione per evitare problemi di sincronizzazione.

Twelve-factor favorisce fortemente i linguaggi che forniscono una shell REPL pronta all'uso e che semplificano l'esecuzione di script una tantum. In una distribuzione locale, gli sviluppatori richiamano processi di amministrazione una tantum tramite un comando shell diretto all'interno della directory di estrazione dell'app. In una distribuzione di produzione, gli sviluppatori possono utilizzare ssh o un altro meccanismo di esecuzione di comandi remoti fornito dall'ambiente di esecuzione della distribuzione per eseguire tale processo.
